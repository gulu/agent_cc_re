## 2. 技术债务

| 债务 | 影响 | 优先级 | 预估工时 | 计划修复 |
|------|------|--------|:--:|----------|
| 规则引擎 `string.Contains()` 粗糙匹配 | 误报率 ~40% | **P0** | 4d | Phase 1（NegationDetector + 知识库清洗） |
| 知识库硬编码在 DbSeed.cs | 难维护，非程序员无法修改 | **P0** | 2d | Phase 1（YAML 外置） |
| TinyBERT 通用模型无医学微调 | 精度不足，NER/NLI 效果差 | **P0** | 3d | Phase 1（MedBERT 替换 + ONNX 导出） |
| 管线伪并行（实际顺序执行） | 延迟浪费 2-3x | P1 | 3d | Phase 2（`Channel<T>` + `Task.WhenAll`） |
| 及时性维度虚设 | 无实际规则 | P1 | 1d | Phase 2 |
| 单元测试覆盖率 < 5% | 回归风险高，重构无安全保障 | P1 | 5d | Phase 2（随重构逐步补齐） |
| 无 CI/CD | 部署靠手动，无自动化质量门禁 | P2 | 3d | Phase 3（GitHub Actions） |
| 无 LLM 效果基线 | 无法量化 Skill Squad 的准确提升 | P1 | 2d | Phase 1 末（200+ 报告基准测试） |

## 3. 上下文锚点

### 3.1 业务规则

- 质控流程：Pre-filter(CPU) → BERT(GPU) → Skills(GPU,按需) → QA(GPU) → Scoring(CPU)
- 误报优先于漏报：`soul.md` 规定"宁可少报，不可误报"
- 冲突消解：规则引擎 < BERT < LLM Skill < Hermes QA
